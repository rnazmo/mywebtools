# ADR (mywebtools)

## ADR-008: Git のブランチ戦略の検討 → main + feature ブランチを採用

- 日付: 2025-04-18
- 状態: 決定済み

### 背景

- 個人開発であるため main ブランチ一本でも運用できる
- ツール単位で開発が分かれる性質を考慮すると、feature ブランチの採用が適している

### 決定内容

- main + feature ブランチ戦略を採用する。

### 理由

- ツール1本の追加がPR1本に対応し、変更の意図がコミット履歴に残りやすい
- 自分の開発スキル向上のために、main + feature ブランチでの開発に慣れたい
- フロントエンドの開発フローの流れを掴む上でも適している

### トレードオフ

- main ブランチ一本に比べてブランチ操作の手間が増える
- 個人開発のため厳密に運用しなくてよい

## ADR-007: テストの導入を検討 → テストは後回しにする

- 日付: 2025-04-18
- 状態: 決定済み

### 背景

- フロントエンドのテストに不慣れであり、学習コストが高い。
- そもそもフロントエンドに不慣れである。
- 一方で「動くものを作る」ことを優先したい。
- フロントエンドのテストの種類:
    - ユニットテスト
        - 関数・コンポーネント単体が正しく動くかを確認する
        - ツール: Vitest（Vite と相性が良い）
    - インテグレーションテスト  
        - 複数のコンポーネントが組み合わさったときの動作確認
        - ツール: Testing Library（React Testing Library）
    - E2Eテスト（End to End）
        - ブラウザを実際に動かして、ユーザー操作を再現する
        - ツール: Playwright, Cypress

### 決定内容

- 当面はテストなしで進める。
- 将来 Vitest + React Testing Library を導入することを検討している

### 理由

- テストの学習で躓くよりも、まず動くものを作ることを優先する
- Vite プロジェクトとの相性から、導入時は Vitest + React Testing Library が自然な選択になる

## ADR-006: リンター・フォーマッターの検討 → Biome を採用

- 日付: 2025-04-18
- 状態: 決定済み

### 背景

- コードの品質維持・スタイル統一のためにリンターとフォーマッターの導入を検討した。
- 役割の違い:
    - リンター   → コードの「問題」を検出する（バグになりそうな書き方など）
    - フォーマッター → コードの「見た目」を統一する（インデント・改行など）
- 選択肢:
    - ESLint + Prettier
        - 現状のフロントエンドのデファクトスタンダード
        - 設定ファイルが2つ必要
        - 設定が競合することがある
    - Biome
        - 1つでリンターとフォーマッターの両方を担える
        - 高速
        - 設定がシンプル

### 決定内容

Biome を採用する。

### 理由

- ESLint + Prettier は、過去に設定の競合などのトラブルを経験していて、良い印象が無い
- ESLint + Prettier は、設定ファイルが2つ必要で構成が煩雑になる
- Biome は設定ファイルがシンプルになる
- Biome は学習コストもあまり高くなさそうで、導入が楽そう
- 根拠は薄いが、Biome が今後のスタンダードになりそう

### トレードオフ

- Biome は ESLint + Prettier に比べて情報量が少ない
- Biome はまだ枯れていない可能性
    - ただし ESLint + Prettier でも十分にトラブルが発生していたため、許容できるデメリットだと判断した

## ADR-005: パッケージマネージャーの検討 → npm

- 日付: 2025-04-18
- 状態: 決定済み

### 背景

- 選択肢：
    - npm    → Node.js に標準添付。
    - pnpm   → 高速・省ディスク。チーム開発や大規模で恩恵が出やすい。
    - yarn   → 昔 npm が遅かった時代に普及。今は優位性が薄れた。
    - bun    → 新興。速い。まだ枯れていない。
- npm と yarn は使用経験有り。ただし、どちらも十分に使い慣れているとは言えない
- pnpm は気になっていて、使ってみたい。

### 決定内容

- npm を使う

### 理由

- npm で十分
- yarn は使う理由が薄い
- pnpm は良さそう。ただし、今の段階では npm との差を実感しにくい
- まずは npm を使って、フロントエンド開発の流れを掴むべき

## ADR-004: Node.js のバージョン管理方法を検討 → mise

- 日付: 2025-04-18
- 状態: 決定済み

### 背景

- Node.js のバージョン管理方法として何を使うか
- 選択肢：
    - mise
    - asdf
    - nvm
    - n, nodenv, volta, ...
- 選択肢は非常にたくさんある
- 使い慣れているのは mise と nvm.
- asdf, mise は、複数言語のバージョン管理を一元化できる

### 決定内容

- mise を使う

### 理由

- mise を別のプロジェクトでも使用していて、使い慣れている
- mise の感触が、かなり良い
    - シンプルに導入・利用ができ、学習コストが低い
    - 機能もシンプル
    - これ一つで多数のツールのバージョンを一元管理できる

## ADR-003: カスタムドメイン採用の有無を検討

- 日付: 2025-04-18
- 状態: 決定済み

### 決定内容

- （少なくともしばらくの間は）カスタムドメインは採用しない

### 理由

- まずは基本機能の作成に集中するべき

## ADR-002: Cloudflare Pages は将来別プロジェクトで試す

- 日付: 2025-04-18
- 状態: 決定済み

### 背景

- Cloudflare Pages は Vercel と同等の機能を持ち、無料枠が広い。
- スキル向上のために触っておきたいサービスの一つ。

### 決定内容

- 今回の mywebtools では採用しない。将来別プロジェクトで試す。

### 理由

- 今回はフロントエンドスキルの習得を優先するため、ツールの学習コストを分散させない
- まずは Vercel に十分に慣れたべき。そうして始めて Cloudflare Pages との比較ができるようになる

## ADR-001: デプロイ先の検討 → Vercel を採用

- 日付: 2025-04-18
- 状態: 決定済み

### 背景

- 静的サイトのホスティング先を検討した
- 検討した選択肢：
    - GitHub Pages
        - 良い点：無料；GitHub と一体なのでシンプル
        - 悪い点：React Router を使う場合、追加設定が必要（SPAのリダイレクト問題）；ビルド設定がやや面倒
        - 慣れている
    - Vercel
        - 良い点：設定がほぼゼロ；push したら自動デプロイ；SPAのルーティング問題を勝手に解決してくれる；プレビューデプロイ機能がある
        - 悪い点：外部サービスへの依存が増える；無料枠に制限あり（個人利用なら実質問題なし）
        - 多少使ったことがある。慣れたい。
    - Cloudflare Pages
        - Cloudflare のサービスの一つで、静的サイトのホスティングができる
        - Vercel の競合。Vercel とほぼ同等。
        - 無料枠が Vercel より広め。
        - 使ったことがない。使ってみたい。
- SPAのルーティング問題とは
    - /stopwatch などに直接アクセスしたときに 404 になる問題。
    - GitHub Pages はそのままでは対応していないので、404.html を使ったハックが必要になる。
    - Vercel はこれを自動で処理するので楽。

### 決定内容

- Vercel を採用する。
- Cloudflare Pages は将来別プロジェクトで試す（ADR-002 参照）

### 理由

- GitHub Pages を選ばない理由：
    - 既に慣れているので、技術的に得るものが少ない
    - 特定プラットフォームへの依存度を下げるため、GitHub に強く依存することを（無理のない範囲で）避けたい
- Vercel を選ぶ理由：
    - 設定がほぼゼロで、フロントエンド開発本体に集中できる
    - フロントエンド業界で標準的なサービスであり、触っておく必要がある
    - SPA のルーティング問題（直接URLアクセス時の404）を自動で解決してくれる
- Cloudflare Pages を選ばない理由：
    - Cloudflare Pages も同等の機能を持つが、今回は学習コストの分散を避けるため見送った
